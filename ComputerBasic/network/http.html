# HTTP

# HTTP

http是一个简单的请求-响应协议，基于TCP/IP协议之上的应用层协议 ###特点

# 状态码

状态码 描述
100 Continue 继续，客户端应继续其请求
101 Switching Protocols 切换到更高级的协议，比如WebSocket
200 OK
201 Created 已创建。成功请求并创建了新的资源
202 Accepted 已接受。已经接受请求，但未处理完成
301 永久重定向 被请求的资源已永久移动到新位置
302 暂时重定向
303 与301类似。使用GET和POST请求查看
304 资源未修改 触发协商缓存
305 Use Proxy 使用代理。所请求的资源必须通过代理访问
307 临时重定向。与302类似。使用GET请求重定向
400 Bad Request 请求语法错误，服务器无法理解
401 Unauthorized 请求要求用户的身份认证
403 Forbidden 服务器理解请求客户端的请求，但是拒绝执行此请求
404 Not Found
500 服务器内部错误
501 Not Implemented 服务器不支持请求的功能，无法完成请求
502 作为网关或者代理工作的服务器尝试执行请求时，从远程服务器接收到了一个无效的响应
503 由于超载或系统维护，服务器暂时的无法处理客户端的请求
504 Gateway Time-out 充当网关或代理的服务器，未及时从远端服务器获取请求
505 HTTP Version not supported 服务器不支持请求的HTTP协议的版本，无法完成处理

# HTTP请求方式

请求方式 描述
GET
POST 向指定资源提交数据
HEAD 获取报头
PUT 浏览器向服务器发送的数据取代指定的文档内容
DELETE 请求服务器删除指定页面
CONNECT HTTP1.1协议中预留给能够将连接改为管道方式的代理服务器
OPTIONS 允许浏览器查看服务器性能
TRACE 显示服务器收到的请求，用于测试&诊断

# HTTP2

  1. 二进制协议（而非文本格式）引入二进制数据帧和流的概念，其中帧对数据进行顺序标识，
  2. 多路复用（并行，不需要像1.x得到响应后才能发下一个请求
  3. 对数据包做标记（保证顺序 不会出现数据错乱）
  4. 头信息压缩（客户端和服务器同时维护一张头信息表，同样字段只发送索引号）
  5. 服务器推送（只通过一次HTML请求，就同时把样式、图片等全部发送给浏览器，通过一次通信获得所有资源。）

# HTTP1.0&HTTP1.1

  1. HTTP1.1引入持续连接的概念,在一个TCP连接上可以传送多个HTTP请求与响应，减少了建立关闭连接的消耗和延迟
    默认开启Connection:keep-alive
  2. 多了缓存处理Etag,

# HTTPS

对一个密钥进行非对称加密，然后使用这个密钥进行对称加密

# HTTPS与HTTP区别

  1. 端口号：8080 443
  2. 安全性：HTTP明文传输 HTTPS在HTTP和TCP间添加安全协议SSL/TSL加密

###【请求过程4次握手】

  1. 浏览器发HTTPS请求
  2. 服务器有一对密钥 公钥&私钥，私钥保存，将包含公钥的证书发给浏览器
  3. 浏览器解析证书，检测公钥是否合法后，生成密钥（就是一个随机数），用公钥对这个密钥加密
  4. 服务器用私钥对其解密，就获得了密钥。然后使用密钥对数据进行对称加密，发送给浏览器。
  5. 浏览器用密钥解密，得到数据。

# Cookie&Session

# Cookie

服务器通过设置set-Cookie响应头，将Cookie发送给浏览器。浏览器保存在本地，在下次发送请求时带上

# session

  • 存放在服务器，用来存放用户数据
  • 浏览器第一次发请求时服务器自动生成hashtable&sessionid来唯一表示这个hash表，并将sessionid存放在cookie中响应给browser。 第二次请求 将携带着sessionid的cookie发给服务器，通过sessionid找到用户对应的表
  • session依赖于cookie

# 区别

  1. cookie保存在客户端
  2. 安全性
  3. session消耗服务器资源 cookie影响网络性能

# 浏览器缓存

# 强缓存

在客户端完成，不请求服务器

  1. Expires
  2. cache-control
    • max-age
    • public/private 代理服务器是否缓存。
    • immutable 即使用户做了刷新操作，也不向服务器发起http请求
    • no-cache 跳过强缓存，直接协商缓存
    • no-store 取消强缓存&协商缓存

# 协商缓存

需要客户端和服务器两端进行交互

  1. Etag 一个hash，表示唯一
  2. last-modify 文件的修改时间，精确到秒 response header中的etag、last-modified在客户端重新向服务端发起请求时，会在request header中换个key名：
// response header
etag: '5c20abbd-e2e8'
last-modified: Mon, 24 Dec 2018 09:49:49 GMT

// request header 变为
if-none-matched: '5c20abbd-e2e8'
if-modified-since: Mon, 24 Dec 2018 09:49:49 GMT

# 三握四挥

# 三次握手

第一次握手： 客户端给服务器发送一个 SYN 报文。 第二次握手： 服务器收到 SYN 报文之后，会应答一个 SYN+ACK 报文。 第三次握手： 客户端收到 SYN+ACK 报文之后，会回应一个 ACK 报文。

# 四次挥手

第一次挥手： 客户端发送FIN报文（进入FIN_WAIT_1状态） 第二次：服务器返回ACK报文，此时还有未发送完的数据（CLOSE_WAIT状态） 第三次：服务器发送FIN报文，请求关闭连接（LAST_ACK） 第四次：客户端发送ACK（TIME_WAIT）

# 问题

  • 为什么建立连接只用三次握手，而断开连接却要四次挥手？
    服务端的FIN和ACK需要分开发，并不是像三次握手中那样，SYN可以和ACK同步发，所以就需要四次挥手
  • 在四次挥手中，客户端为什么在TIME_WAIT后必须等待2MSL时间呢？
    为了保证客户端发送的最后一个ACK报文段能够到达服务端。
    这个ACK报文段有可能丢失，所以若没收到第四次的ACK，会重复第三次的操作重传FIN报文

# TCP滑动窗口

  1. 作用：流量控制&乱序重排
  2. 是什么：告诉发送端一次可以发送的数据大小 接收端将自己可以接收的缓冲区大小放到TCP首部的窗口大小字段，通过ACK告诉发送端 窗口大小越大说明网络的吞吐率越高

# 拥塞控制

网络发生拥堵时，会产生数据包时延、丢失的情况，这时若是重传会使网络负担更加严重。所以TCP会自我牺牲，降低发送的数据量。
